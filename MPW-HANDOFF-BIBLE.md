# 47. mpw_bible_writer.py v4.0 — FULL SPEC — TWO-PASS ARCHITECTURE

**Built Session 29. 55-check validation suite passes.**

## Why v4.0

v3.0 produces good entries. v4.0 produces entries nobody can match. The target is 6,000 words per entry with content components that don't exist anywhere else on the internet.

## Word Count Target

- Floor: 5,500 words
- Target: 6,000 words
- Ceiling: 6,500 words

## Two-Pass Architecture

**Pass 1 (20,000 tokens) — Structured Data Pass**

Returns JSON only. Contains all structured, factual, schema-critical content:
- term, slug, category, part_of_speech, pronunciation, entry_pos_label
- meta_description (155 chars, SEO-optimized)
- meta_keywords (6-8 terms)
- quick_answer (under 40 words, featured-snippet engineered)
- excerpt (under 120 chars for bible-index.json)
- emotional_hook (one italic pull-quote)
- svg_diagram (viewBox, title_id, title_text, svg_content)
- parameters (minimum 4 for signal processing, minimum 3 for others — each with name, title, desc in technical AND perceptual language)
- quick_reference_rows (minimum 5 rows — columns adapt to category)
- quick_reference_cols (column headers — adapt to term)
- quick_reference_note
- copy_table_text
- types (minimum 3 — name, hardware, desc)
- related_terms (minimum 4 — slug, term, preview)
- further_reading_slugs (6-10 confirmed MPW article slugs)
- faq (8 questions — mix beginner/intermediate/advanced)
- entry_tags (5 tags)
- schema_about_same_as
- youtube_embed (video_id or null — ONLY if 100% certain)
- spotify_embeds (track_uri or null — for link generation only, NOT iframes)
- calculator_type (bpm-timing | frequency-chart | gain-calculator | none)
- word_count_estimate
- read_minutes
- the_number
- producer_quote
- signal_chain_position
- hardware_vs_plugin_rows
- genre_application_rows
- interaction_warnings
- red_flags
- green_flags

**Pass 2 (16,000 tokens) — Prose Pass**

Receives Pass 1 JSON as context. Returns JSON with all long-form prose fields:
- definition_html (5-6 paragraphs — opens with emotional hook — final paragraph = why_it_matters_now)
- how_it_works_html (3-4 paragraphs — technical + perceptual language)
- history_html (4-5 paragraphs — EVERY paragraph names specific engineer, hardware, recording, studio, year)
- how_producers_use_it_html (4-5 paragraphs by instrument/context + DAW grid)
- track_examples (minimum 5 — different artist, genre, era — each with timestamp + listening_guide)
- mistakes (minimum 6 — each names specific Hz/dB/ms/ratio values)
- common_misconception_html
- producers_verdict_html
- progression_path_html
- before_after_text
- section_summaries

## New Content Components — JSON Schemas

### the_number
```json
"the_number": {
  "value": "6 dB",
  "label": "Maximum gain reduction before over-compression becomes audible",
  "context": "One sentence on when this rule applies and when to break it."
}
```
HTML: `<div class='the-number-box'>` — amber box at top of Quick Reference section.

### common_misconception_html
```json
"common_misconception_html": "<div class='misconception-block'><span class='misconception-label'>Common Misconception</span><p class='misconception-belief'>Most producers believe...</p><p class='misconception-reality'>Here's what's actually happening...</p></div>"
```

### producers_verdict_html
```json
"producers_verdict_html": "<div class='producers-verdict'><span class='verdict-label'>The Producer's Verdict</span><p>For most producers, start with: [specific recommendation]</p><p>If you're doing [specific context], use [specific alternative]</p><p>Avoid [specific thing] until you understand [prerequisite]</p></div>"
```
Placed after Types section. Opinionated. Takes a position. No hedging.

### red_flags / green_flags
```json
"red_flags": [
  "If your kick loses punch when you add compression, your attack is too fast — try 20-30ms"
],
"green_flags": [
  "Gain reduction meter never exceeds 6dB consistently on individual tracks"
]
```
HTML: `<div class='red-green-flags'>` — visual indicators, placed after Common Mistakes.

## System Prompt (applied to BOTH passes)

```
You are writing an entry for The Producer's Bible — MusicProductionWiki.com's authoritative music production reference.

The Producer's Bible is to music production what Britannica is to general knowledge. It is not a blog, not a listicle, not a beginner's guide. It is a reference work that a professional music engineer would trust.

Voice: Authoritative. Direct. Knowledgeable. Not academic — written by someone who has spent thousands of hours in a studio, not in a university. The reader is a serious producer who will immediately know if the content is superficial.

Standards: Every claim is specific. Every technique has a number attached. Every track example names the exact moment and what to listen for. Every mistake names the exact symptom and the exact fix.

This entry will be read by producers at every level — beginners who are learning, professionals who are checking a reference, and everyone in between. Beginners need the foundation. Professionals need the depth. Both need the precision.

You are not writing marketing copy. You are not hedging. You are not being safe. You are telling producers exactly what they need to know, in the most useful format possible.
```

## Quality Rules (enforced in Pass 1 prompt)

```
QUALITY RULES — NON-NEGOTIABLE:

TERM: {term} — every section must be specifically about {term}, not a generic music production concept.

emotional_hook: One italic pull-quote. Must NOT say "{term} is the most important tool in music production." Must create genuine curiosity or reframe the producer's relationship with the concept.

parameters: Each parameter description MUST include technical language AND perceptual language. Separated clearly. Include specific numbers.

track_examples: minimum 5. Each MUST be a DIFFERENT artist from a DIFFERENT genre from a DIFFERENT decade. No two examples from the same artist. No two from the same genre. The note must be specific enough that a producer could listen to the track and find the moment described.

faq: 8 questions. Must mix: 2 beginner (What is..., How do I start...), 3 intermediate (technique-specific, comparison-based), 3 advanced (nuance, edge cases, professional context). Answers: 3-5 sentences each. Direct. Opinionated where appropriate. Never "it depends" without then answering it.

history_html: EVERY paragraph must name: one specific person, one specific piece of hardware or recording, one specific studio or location, one specific year. No paragraph may open with generic historical language.

mistakes: minimum 6. Each must name: a specific audible symptom, a specific cause (which parameter, which value), and a specific fix.

red_flags: 5 minimum. Each must name a specific audible symptom with approximate parameter values where possible.

green_flags: 5 minimum. Each must name a specific audible or measurable result that confirms correct application.

interaction_warnings: minimum 2. Each must name: the interacting tool/process, the problematic condition, the audible result, and the fix.

producer_quote: Only include if Claude is confident the quote is real and accurately attributed. Omit entirely if uncertain. No fabricated quotes.

the_number: Must be specific, session-usable, and defensible. One value. Not a range.

genre_application_rows: Values must be genuinely different across genres.

hardware_vs_plugin_rows: Must reflect genuine character differences, not marketing copy.

TOKEN BUDGET: Pass 1 = 20,000 tokens. Pass 2 = 16,000 tokens. Never let one field consume more than 15% of budget. A truncated JSON is worthless.

NEVER use placeholder text in any field.
NEVER generate a quote you are not confident is real.
NEVER write "it depends" as an FAQ answer without then answering what it depends on.
```

## File Architecture (v4.0)

```

call_claude_pass1(prompt, label) → 20,000 tokens, system prompt
build_pass2_prompt(slug, term, category, pass1_json) → prose prompt with Pass 1 context
call_claude_pass2(prompt, label) → 16,000 tokens, system prompt

build_catalog_content(slugs, bible_slugs) → regenerates MPW-CATALOG.md
```
